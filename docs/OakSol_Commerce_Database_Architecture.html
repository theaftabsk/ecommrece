1. Overview & Isolation Strategy

The OakSol Commerce database is architected to support enterprise-grade SaaS operations from day one. By adopting a shared-database, shared-schema pattern with UUID primary keys and mandatory shop_id foreign key columns, the application achieves robust logical isolation. This design prevents cross-tenant data leaks and eliminates the need for expensive database redesigns when transitioning from a single-store pilot (Phase 1) to a multi-merchant SaaS platform (Phase 2).

Security Compliance: All tables carrying tenant data reference shops(id). Rows are queried using explicit tenant filtering. API keys and payment secrets are configured to be encrypted at rest in the database layer.

10. Indexes & Performance

Every query in a multi-tenant system must be prefixed by shop_id. The 16 indexes below ensure PostgreSQL never performs a full table scan, maintaining sub-millisecond response times even at high transaction volumes.

Index Name On Table & Columns Purpose
idx_products_shop products(shop_id) Multi-tenancy catalog lookup
idx_orders_shop orders(shop_id) Multi-tenancy sales lookup
idx_customers_shop customers(shop_id) Multi-tenancy buyer lookup
idx_variants_product product_variants(product_id) Loads SKUs for specific product catalog page
idx_variants_sku product_variants(sku) Product scanning/SKU lookups
idx_cart_items_cart cart_items(cart_id) Retrieve lines within active cart session
idx_order_items_order order_items(order_id) Line item lookups for invoices
idx_payments_order payments(order_id) Gateway response webhook mapping link
idx_inventory_variant inventory_logs(variant_id) Stock ledger adjustments tracking
idx_reviews_product reviews(product_id) Fetch star ratings for product pages
idx_notifications_cust notifications(customer_id, is_read) Query unread alerts for inbox
idx_activity_shop_time activity_logs(shop_id, created_at DESC) Admin action chronological audit logs
idx_shop_domain shop_domains(domain) UNIQUE Every web request mapping — crucial for multi-tenant routing

12. Key Design Decisions

The architectural features below distinguish OakSol Commerce's database design:

UUID Primary Keys

All tables use gen_random_uuid(). This guarantees distributed database compatibility, makes offline UUID generation safe, and prevents URL enumeration attacks.

Multi-Tenant Isolation

Every tenant table contains a shop_id foreign key. Access pipelines query using explicit tenant IDs, ensuring zero possibility of cross-tenant data leaks.

Custom Domain Routing

The shop_domains table stores subdomains and custom domains. The middleware maps incoming hostnames to corresponding shop_id root entries in sub-milliseconds.

Audit Log Trail

The inventory_logs and activity_logs tables track every stock adjustment and dashboard modification, ensuring full accountability.

Order Snapshot Pattern

To prevent historical records from breaking when products change, order_items.product_snap caches product details as JSONB at purchase time.

Address Snapshotting

Similarly, orders.shipping_address stores shipping addresses as JSONB. Customers can modify their profile addresses without breaking historical order details.

Guest Session Lifecycle

Using customers.guest_token and carts.session_id, shoppers can complete guest checkouts. These guest records can easily merge with new accounts later.

TimeZone Compliance

All timestamps use TIMESTAMPTZ. This ensures accurate tracking across international regions and accounts for Daylight Saving Time (DST).

Soft Delete Policy: To maintain historical integrity and prevent database constraint issues, the platform utilizes status codes (such as active, archived, suspended, draft) instead of hard deletes.
